# エージェント行動規範 (Workspace Rules)

## 後工程はお客様
*   **ルール**:
    *   すべてのエージェント（リード、開発、検証、調査等）は、自分の作業が完了した際、または他からの情報提供を待つ状態（ブロック状態）になった際、自身の対話スレッドで独り言をつぶやいて待機するだけの状態にしてはなりません。
    *   作業完了、あるいはブロックされた場合は、必ず「後工程（呼び出し元のメインエージェント、あるいは最終決定権を持つユーザー）」に対して、**完了報告**または**明確なアクション要求/質問**を通知・送信してください。
    *   次のプロセスを担う人が「何を待っているのか」「何が終わったのか」がひと目でわかるよう配慮し、作業がスタックするのを防いでください。
    *   次のエージェントが待っていることを強く意識してください。
    *   次のエージェントが期待している成果物を、自分の権限内である程度意識してください。ただし、自分だけではわからない情報もあるため、あまり自分だけで抱え込んで考えすぎず、必要に応じて速やかに前工程やユーザーに相談・確認を行ってください。

## メイン環境のブランチ固定（Checkout禁止）
*   **ルール**:
    *   メインの分析・実行環境である `/Users/masaru/Projects/Trading` において、`git checkout` 等でブランチを `main` 以外に切り替えてはなりません。ユーザーがいつでも本番用の分析を実行できる状態を維持するためです。
    *   コードの変更や開発ブランチでのコミット作業は、必ずサブエージェントに独立したワークスペース（worktree等）で実行させるか、一時的なワークツリーを別フォルダに作成して行ってください。

## トークン消費削減の徹底
*   **ルール**:
    *   **思考と回答の簡略化**: エージェントの思考プロセス（Thinking）は、現状の課題と次の一手を1〜2文程度で短く整理するだけに留め、長々と書かないこと。回答は結論ファーストとし、マークダウンの箇条書きなどを用いて簡潔にまとめること。
    *   **部分読み込みの徹底**: ソースコードやログファイルなど、100行を超えるファイルを閲覧（view_file）する際は、必ず `StartLine` と `EndLine` を指定し、必要な部分のみを最小限ロードすること。ファイル全体を一度に読み込んで履歴を肥大化させてはならない。
    *   **サブエージェント呼び出しの最適化 (改善案3)**: 試行錯誤を伴うコード実装や検証はサブエージェントで行い、`git revert` や `git push`、単純なファイル移動などの「一発で確実に終わる軽微な操作」はメインエージェントが直接行うこと。

## カスタム対話ルール
*   **思考プロセスの明示**:
    *   ユーザーから理由を尋ねられた場合は、必ず回答に至る「思考プロセス（論理ステップ）」を明示して説明してください。
*   **本質的な相談への注力**:
    *   相談は基本的に本質的な内容（コアロジックや大前提の考え方など）に絞ります。例外ケースや現実で発生し得る様々な個別のケース（極端な市場の歪みや特異な事例など）は、ユーザーから個別に指示がない限り無視して回答を構成してください。
